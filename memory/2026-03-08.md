## Temporary Workaround Applied

Changed vetoProposal to split conditions to guarantee subtraction only evaluated when `block.timestamp < eta`:
```solidity
require(block.timestamp < eta, "Governor: veto only allowed within 48h of execution");
require(eta - block.timestamp <= VETO_WINDOW, "Governor: veto only allowed within 48h of execution");
```
First require ensures `eta > block.timestamp` holds before second require executes, preventing underflow.

## Root Cause & Fix (2026-03-08 05:20 GMT+4)

The failure in `test_VetoFailsTooEarly` was due to **test underflow**, not contract code.

Test logic:
- Timelock delay = 2 days (172,800 seconds).
- After queue, `eta = block.timestamp + 172800`. In the test, `block.timestamp` at queue time is 1 (the default), so `eta = 172801`.
- Test attempts `vm.warp(eta - 72 hours)`. Since 72 hours = 259,200 seconds, this calculates `172801 - 259200`, which underflows to a huge positive number (type(uint256).max - 86399). The result is a timestamp far in the future.
- Then `vetoProposal` checks `block.timestamp < eta` (false, because warped timestamp is huge), and then executes `eta - block.timestamp` which underflows, causing the panic.

**Fix:** Ensure the base timestamp before queue is sufficiently high so that `eta - 72 hours` remains positive. We modified `test_VetoFailsTooEarly` to set `baseTimestamp = 1_000_000` before any proposal steps. This yields `eta = 1,000,000 + 172800 = 1,172,800`, and `eta - 72h = 913,600` (valid). Test now passes.

The two-step require is correct and safe when inputs are non-negative.
